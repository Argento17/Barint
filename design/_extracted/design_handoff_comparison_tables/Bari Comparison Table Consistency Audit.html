Q1 — Are there differences between products? Yes, three kinds.

There shouldn't be — but there are. Six categories share a "canonical" path, yet that path renders through two unrelated row components (one for mobile, a different one for desktop). On top of that, a v2 layout flag is switched on for only two of the six, so Hummus and Maadanim show a structurally different row than Yogurts, Spreads, Snacks and Bread. And Milk is a completely separate, hand-built table that shares none of the above. Net result: a shopper moving between categories meets up to three different table designs.

02

What's actually different

Four concrete sources of divergence surfaced by the scan, ordered by how visible they are to a shopper comparing categories.

D1

Milk is a parallel table

High · shopper-visible

Milk renders through its own MilkComparisonPage + ProductShelfRow and shares no row, thumbnail, grade or expansion component with the other six. It shows a 4-cell metric grid (protein · sugar · additives · main ingredient) and an expansion taxonomy — מה חשוב לדעת / מה מעלה / מה מוריד / בהשוואה + an advanced pillars panel — that exists nowhere else. Sugar-per-100ml and "main ingredient" are surfaced only here.

Milk
D2

Mobile and desktop are different components — everywhere

High · structural

Across all six canonical categories, the desktop table is BariProductShelfRow and the mobile table is ProductRow — two hand-maintained components. They diverge in grade UI (BariGradeBadge vs ScoreChip), thumbnail (BariProductThumbnail vs ProductImage), and expansion (inline markup vs the ExpansionSection component). The "בקצרה:" prefix appears on desktop only; mobile shows the same line unlabelled. Any fix to one must be hand-ported to the other.

All 6 canonical
D3

The v2 row ships to 2 of 6

Medium · shopper-visible

A v2Slice flag is true only in HUMMUS_V2_SLICE and MAADANIM_V2_SLICE. Where it's on, the row shows strongest +/− reasons, a promoted confidence indicator, an inline protein metric, and a single expansion disclosure. Where it's off (Yogurts, Spreads, Snacks, Bread) the row still shows the old insightLine, hides confidence in a 10px footnote, and uses a two-level "advanced" toggle. The spec calls this a "proof slice" — it was never rolled out.

Hummus ✓Maadanim ✓Yogurts ✗Spreads ✗Snacks ✗Bread ✗
D4

A whole desktop table path ships dead

Low · maintenance risk

ProductRow contains a complete desktop web-grid branch (rank · image · name · score columns) and ProductTableHeader renders the column guide — but no shipped route passes layout="web" to ComparisonShelfPage; every page wraps it in lg:hidden and uses BariComparisonDesktopPage instead. So this path renders nothing in production yet still has to be reasoned about on every change.

03

Q2 — Improvements I'd suggest

The goal is one table experience that every category inherits for free, with per-category data — not per-category layout. Ordered roughly by leverage.

IMP-1

Collapse to one responsive row component

High effort · highest leverage

Merge BariProductShelfRow and ProductRow into a single component that handles mobile and desktop with CSS breakpoints, and delete the loser. This removes D2 at the root — one place to change grade UI, thumbnails, and the expansion. Pick one grade primitive (BariGradeBadge) and one thumbnail (one of the three current ones) as part of the merge.

FIXES D2 · also unblocks D3 (one row to flip), D4 (the dead path goes with it)

IMP-2

Make v2 the default, then remove the flag

Low effort · high visibility

Flip v2Slice on for all six categories (or invert the default to true), QA the four that lag, then delete the flag entirely so there is no "v1 vs v2" fork to drift again. This ends the Hummus-vs-everyone-else gap that a shopper sees immediately.

FIXES D3 · depends on IMP-1 landing first for a clean single flip

IMP-3

Fold Milk into the shared system

High effort · removes a fork

Milk's extras are data, not layout: sugar-per-100ml and main-ingredient become optional metrics fields on the shared VM, rendered (or hidden) by the one row component. Map its expansion taxonomy onto the shared section identities so Milk reads like every other category. This retires the largest single source of divergence.

FIXES D1 · should follow IMP-1 so there is one target to fold into

IMP-4

Finish the v2 spec — uniformly

Medium effort · per spec

Even the "v2" categories only got ~3 of the 9 specced items. The aligned metric column (protein · additives · base %), the score-band jump rail, in-list band dividers, the header score histogram, and the density toggle are all still unbuilt. Build them once in the shared component so all categories gain them together — the aligned metric column is the actual differentiation mechanism the spec calls for.

FIXES spec §1–4, §7, §9 · build on the merged component from IMP-1

IMP-5

Delete the dead web-grid path

Low effort · cleanup

Remove ProductRow's unused web branch, ProductTableHeader, and ComparisonShelfPage's web layout mode — or wire them up. Right now they are pure cognitive overhead and a trap for the next engineer who edits the row. (If IMP-1 lands, this falls out for free.)

FIXES D4

IMP-6

Lift category-wide notes out of the rows

Low effort · per spec §8

Per the spec, category-level disclosures (e.g. the Hummus "fat values not shown" note, relativity caveats) belong once in the header or methodology block — not repeated in every row's unknowns. This both de-clutters the row and makes per-row unknowns mean "product-specific" again, consistently across categories.

FIXES spec §8 · independent of IMP-1, can ship anytime
